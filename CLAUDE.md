# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A take-home test task: a browser-based, non-destructive image editor (Vue 3 + Vuetify 3 + Pinia + TypeScript). The brief (full text in `README.md`) explicitly cares more about how edits are *modeled* than about UI polish — reviewers are expected to read the Pinia store and the `src/utils/*` pure functions. `NOTES.md` is the submission write-up explaining the design decisions (operations-as-data model, why CSS/canvas filters, JSON replay shape, why dark theme, bonus scope) — read it before making architectural changes so new code stays consistent with the rationale already given.

## Commands

- `npm i && npm run dev` — install and start the Vite dev server (this is the required "it must run" command from the brief).
- `npm run build` — type-checks (`vue-tsc -b`) then production-builds (`vite build`). This is also the type-check command; there is no separate `typecheck` script.
- `npm run preview` — serve the production build locally.
- `npm test` — runs the Vitest suite once (`vitest run`); `npm run test:watch` for watch mode.

There is no lint script. `tsconfig.app.json` has `noUnusedLocals`/`noUnusedParameters` on, so `npm run build` will fail on unused variables/params — treat that as the correctness gate. `tsconfig.app.json`'s `include` also covers `tests/**/*.ts`, so the test files are type-checked by `npm run build` too, even though Vitest itself doesn't type-check.

No path aliases are configured; all imports are relative.

## Tests (`tests/`)

Vitest, configured via `vitest.config.ts` (`environment: 'node'` — no jsdom; Node 24's native `File`/`URL.createObjectURL`/`crypto.randomUUID` are enough for the store + pure-util tests, so no DOM polyfill is needed). Test files live in a top-level `tests/` folder that mirrors `src/`'s shape (`tests/utils/*.test.ts`, `tests/stores/*.test.ts`), not co-located with the source — imports reach back into `../../src/...`.

Coverage is scoped to the store and `src/utils/*` per the brief's own framing (see "What this is" above). `export.ts` is the one piece deliberately left untested — `downloadImage` calls real `Image.decode()`/`OffscreenCanvas#convertToBlob`, neither of which exists in Node without a canvas polyfill, and it's thin orchestration over `drawPipeline` (which *is* tested) plus a `triggerDownload` DOM side effect.

- `tests/utils/transform.test.ts` — `isNeutralTransform`, `normalizeRotation` (wraparound + float-drift rounding), `rotatedDimensions`.
- `tests/utils/filters.test.ts` — `isNeutralAdjustments`, `toCssFilter`, `applyAdjustmentsToPixels` (identity/brightness/greyscale math, plus the chunked-vs-single-pass branch — stubs `requestAnimationFrame` and checks pixels at both ends of a >4MP array to confirm the chunk boundaries are covered).
- `tests/utils/render.test.ts` — `drawPipeline`, with `globalThis.OffscreenCanvas` stubbed to a duck-typed `FakeOffscreenCanvas`/`FakeContext` pair that records every `translate`/`rotate`/`scale`/`drawImage`/`getImageData`/`putImageData` call instead of actually rendering. Covers: full-image draw + baked CSS filter when `ctx.filter` is supported, crop rect used as both source-rect and output size, rotate+flip going through an intermediate canvas before the final draw (asserts the exact `translate`/`rotate`/`scale` args), and the `getImageData`/`applyAdjustmentsToPixels`/`putImageData` fallback when unsupported. Since `supportsCanvasFilter()`'s result is memoized at module scope, each test calls `vi.resetModules()` and dynamically re-`import()`s `render.ts` so the filter-supported flag (a module-level test variable the `FakeContext` setter reads) actually takes effect.
- `tests/utils/operations.test.ts` — `toOperations`/`fromOperations` round-trip, and `parseOperationsFile`'s validation (bad JSON, wrong version, unknown type, non-right-angle rotate, duplicate types, non-canonical order).
- `tests/stores/images.test.ts` — `addImage`/`removeImage`/field setters/`applyOperations`/`resetEdits`. Mocks `URL.createObjectURL`/`revokeObjectURL` rather than using Node's real implementation — the real one registers Blobs in a registry that Node's GC can auto-revoke via a `FinalizationRegistry` at unpredictable times, which made spying on the real `revokeObjectURL` flaky across tests.

## Core architecture: operations as data

The central design constraint is non-destructive editing. Nothing about an uploaded image is ever mutated in place; instead the app accumulates *parameters* describing edits, and rendering (preview and export) is a pure function of `(original, parameters)`.

- **`src/types/`**: the shared domain model, split by concern — `adjustments.ts` (`Adjustments`, `FilterName`), `transform.ts` (`Transform`, `RotationDegrees`), `operations.ts` (`Operation` and its `TransformOperation`/`CropOperation`/`AdjustOperation`/`FilterOperation` variants, `OperationsInput`), `image.ts` (`ImageItem`). These are pure type-only files with no runtime exports; every consumer imports the type it needs directly from here rather than from whichever `utils/*`/`stores/*` module happens to also use it — don't reintroduce type re-exports from those modules as a convenience shim. Types that cross-reference each other (e.g. `operations.ts` building on `Transform`/`Adjustments`) import from their sibling file within `src/types/`.
- **Store** (`src/stores/images.ts`, `useImagesStore`): holds `images: ImageItem[]`. Each `ImageItem` is one independent, non-destructive editing session:
  - `file`/`url` — the original upload, an object URL, never rewritten.
  - `transform: { rotate: 0|90|180|270, flipX, flipY }` — rotate is restricted to right-angle multiples (no arbitrary angles) specifically so the crop rectangle stays axis-aligned after rotation. Pipeline order is transform → crop → adjust → filter, so `cropCoordinates` are defined in the *already-transformed* image's coordinate space.
  - `cropCoordinates: Coordinates | null` — from `vue-advanced-cropper`, in the transformed image's coordinate space. Crop never actually cuts pixels; it's only ever used later as the source-rect argument to `drawImage`.
  - `adjustments: { brightness, contrast, saturation }` — a fixed slot, mutated in place by the sliders (not appended to a list).
  - `filter: FilterName | null` — a single nullable slot (`'greyscale' | 'sepia' | 'invert' | 'warm' | 'cool' | 'vintage'`), so only one filter can be active at a time by construction.
  - `resetEdits`/`Reset` just nulls these four slots back to defaults — the original file is untouched throughout.
- **`src/utils/filters.ts#toCssFilter(adjustments, filter)`**: builds one canonical CSS filter string (`brightness() contrast() saturate() [grayscale()|sepia()|...]`), used for the live crop-tool preview (as a CSS `style.filter`) and, via `drawPipeline` below, baked into canvas pixels for thumbnails and export — one function, so preview and export can never drift apart. `applyAdjustmentsToPixels` is the manual pixel-math fallback for browsers where `ctx.filter` is unsupported (Safari < 18); it's `async` and, above `CHUNKING_PIXEL_THRESHOLD` (4,000,000px), processes `ROWS_PER_CHUNK` (200) rows at a time with a `requestAnimationFrame` yield between chunks so a large canvas doesn't block the main thread for seconds in one tight loop — canvases at or under the threshold still run in a single synchronous pass. This is why `drawPipeline`/`downloadImage` are `async` too.
- **`src/utils/transform.ts`**: `defaultTransform`/`isNeutralTransform`/`rotatedDimensions` (swaps width/height for 90°/270°, used by `drawPipeline`'s export-side canvas rotation) and `normalizeRotation` (rounds to the nearest right angle and folds into `0|90|180|270`, used to make sense of `vue-advanced-cropper`'s unbounded accumulated rotate angle — see `EditorPanel.vue` below). `Transform`/`RotationDegrees` themselves live in `src/types/transform.ts`.
- **`src/utils/render.ts#drawPipeline(ctx, image, item)`**: the single rendering function (`async`, see above) — applies `transform` via `ctx.translate`/`rotate`/`scale` onto an intermediate canvas (skipped when transform is neutral), draws the crop as a source-rect from that, then bakes `adjustments`/`filter` via `ctx.filter` (or the manual pixel fallback, `supportsCanvasFilter` feature-detects this once). Used by both `Preview.vue` and `export.ts#downloadImage` — no duplicated drawImage/transform math between preview and export.
- **`src/utils/operations.ts#toOperations(...)`**: derives a serializable, tagged `Operation[]` (`{type: 'transform'|'crop'|'adjust'|'filter', ...}`) from the same store state, using the same "only include non-neutral slots" logic, in pipeline order. This is what gets JSON-exported (the bonus requirement) and is what a replay would run against the original image.
- **`src/utils/export.ts`**: `downloadImage` renders via `drawPipeline` + `convertToBlob` (bakes pixels only here, never during live crop-tool editing) — exports as JPEG (quality 0.92) when `item.file.type === 'image/jpeg'`, PNG otherwise, matching the output extension to whichever branch ran. `downloadOperationsJson` serializes `{version, operations}` via `toOperations`. Both accept an optional `uniqueSuffix` (the caller — `List.vue` — passes the first 6 chars of the item's id when another list item shares the same original filename) inserted before the extension, so two same-named uploads don't overwrite each other's downloaded file.
- **`src/components/Image/Preview.vue`**: the shared preview renderer — calls `drawPipeline` to draw transform+crop+filter onto a `<canvas>`, so it's reused unmodified for both list thumbnails and the full-size preview.

When extending the edit model (e.g. adding a new adjustment or filter), keep the same shape: add a field to `ImageItem` (in `src/types/image.ts`), extend `toCssFilter`/`toOperations`/`drawPipeline` together so preview/export/JSON stay in sync, and don't introduce per-edit mutation of pixel data.

## Component / data flow

The editor is a permanent two-column layout, not a modal: `App.vue` renders an upload/list column next to an always-mounted `EditorPanel.vue`. There is no "open/close the editor" state anymore — only *which* image is active (`editingId`).

- **`App.vue`** is the orchestrator: owns transient UI state (`editingId`, `isNewUpload`, `isEditorLoading`) and wires `useImagesStore` to the panel. It does not hold any editing state itself.
  - `openEditor(id, isNew)` is the only place `editingId`/`isNewUpload` change when switching the active image. It preloads the new image's `url` *before* flipping `editingId` (so the crop tool doesn't flash blank/loading — the same reason the old modal used `eager` + preload-before-open), and if the image being switched *away from* was an unapplied new upload, it removes it from the store first — abandoning it, same as explicitly discarding it. `isEditorLoading` is set around the `preloadImage` await (`try`/`finally`, so it always clears even if decode throws) and passed to `EditorPanel` as `is-loading`, so the panel can show a spinner during that window instead of leaving the previous image (or the placeholder) sitting there looking unresponsive.
  - `removeImage` wraps `imagesStore.removeImage` to also clear `editingId` (back to the placeholder) when the deleted image was the active one.
  - `importOperations` wraps `imagesStore.applyOperations` (called from `List.vue`'s JSON import, which bypasses the panel entirely) with one extra step: if the imported image is the one currently open, it calls `editorPanelRef.value?.reseed()` — `EditorPanel`'s `editingId` watcher only fires on an actual id *change*, so a same-id store update from outside the panel would otherwise leave its draft silently stale. This is the one place `App.vue` reaches into the panel via a template ref instead of props/events; everywhere else stays one-directional.
- **`src/composables/useImageDropZone.ts`**: the drag-and-drop plumbing (dragenter/dragleave depth counter, `dataTransfer.types` filtering, `type.startsWith('image/')` split with rejected names) shared by `App.vue`'s page-wide drop zone and `Uploader.vue`'s drop-zone card — one `onFiles` callback, plus `handleFiles` exposed directly for the `<input type="file">` change handler to reuse the same image-type filtering.
- **`src/components/Image/Uploader.vue`** is a full drop-zone card (not an icon button): click-to-browse or drag-and-drop, accepts multiple files, emits one `select: File[]` event for the accepted batch (rejected names surface in a local `v-snackbar`). Since it's nested inside `App.vue`'s own page-wide drop zone, it wraps the composable's handlers to stop propagation — otherwise a drop on the card would also bubble up and double-add the same files. Its `onFiles` callback (`handleAccepted`) also runs two soft, non-blocking size checks per file before emitting `select` — `file.size` against `MAX_FILE_SIZE_BYTES` (25MB), and, after decoding via a throwaway `Image`/object URL, `naturalWidth * naturalHeight` against `MAX_MEGAPIXELS` (40MP) — surfacing any hits in a separate warning `v-snackbar`. The upload always proceeds regardless; these only warn.
- **`src/components/Image/List.vue`** renders one row per `ImageItem` (thumbnail via `Preview.vue`, edit/download/JSON export+import/delete actions), highlighting the row matching its `activeId` prop. The download and JSON-export buttons each track their own per-item `Set<string>` of in-flight ids (`downloadingIds`/`exportingIds`) bound to `:loading`/`:disabled`, guarded by an early-return re-entrancy check and cleared in `finally` — so a slow export can't be double-triggered and the id is always released even if it throws. `uniqueSuffixFor(item)` (see `export.ts` above) checks the full `images` prop for another item with the same `file.name` before every download/export call.
- **`src/components/Image/EditorPanel.vue`** is a `v-card` (no `v-dialog`) that's always mounted; it renders a loading spinner in `.preview-zone` when the `isLoading` prop is true (see `App.vue#openEditor` above), a placeholder when `src` is `null`, and the full editor otherwise. It holds *draft* copies of transform/crop/adjustments/filter — nothing is written back to the store until **Apply**. Both **Apply** and **Cancel** clear `editingId` back to the placeholder afterward (`App.vue#applyEdits`/`cancelEdit`) — editing one image at a time, then returning to the list, rather than staying parked on the same picture.
  - **Rotate/flip go through the cropper's own API**, not CSS: the rotate/flip buttons call `cropperRef.value?.rotate(±90)` / `.flip(h, v)` (exposed by `Cropper.vue`, which forwards to `vue-advanced-cropper`'s native methods with `{ transitions: false }` — its `.d.ts` doesn't declare that 3rd argument even though the runtime accepts it, hence a cast there). A CSS `transform: rotate()` on the wrapper was tried first and reverted: it doesn't affect layout, so a rotated wide image would visually overflow past the panel into the page below it. The cropper's native rotate handles its own layout instead.
  - **`draftTransform` is a readout, not a target**: `onCropperChange` (bound to the cropper's `@change`) is the *only* place that writes `draftCrop`/`draftTransform`, from whatever `coordinates`/`imageTransforms` the cropper reports — including in response to our own `rotate()`/`flip()` calls. `imageTransforms.rotate` is unbounded/incremental (it's `+=`'d internally), so it's run through `normalizeRotation` before landing in `draftTransform`.
  - **Restoring a saved transform is delta-based**: `rotate()`/`flip()` are both incremental (rotate adds to the current angle, flip toggles per axis), so there's no "set to absolute value" call to make. `applyCropperState(targetCrop, targetTransform)` reads the cropper's last-reported `cropperTransforms` and computes+applies just the delta needed to reach the target — this is what backs `resetDraft` (delta to `defaultTransform`) and image-switch restoration.
  - **Image-switch restoration waits for `@ready`, not the `editingId` watcher**: switching `src` makes the library reset its own internal rotate/flip asynchronously (EXIF-orientation-aware) before it emits `ready` — computing a delta any earlier would race that reset and get silently overwritten. `seedNonCropperDraft` (adjustments/filter/aspect-ratio, no cropper dependency) still runs synchronously off `watch(() => props.editingId, ...)`; `onCropperReady` does the crop/transform restoration once the *new* image is actually settled.
  - **Cancel always returns to the placeholder** (`emit('cancel')` → `App.vue#cancelEdit` clears `editingId`), but only *deletes* the image when it's an unapplied new upload (`isNewUpload`) — labeled "Discard upload" (destructive, `color="error"`) versus plain "Cancel" for an already-saved image (just walks away from the draft; nothing was mutated since draft state never touched the store before Apply).
  - **Fixed-size preview box**: `.preview-zone` (wrapping the cropper, and reused by the placeholder) is `width: 100%; height: clamp(360px, 60vh, 640px)` — a constant box independent of the active image's own dimensions, not an aspect-ratio box. `vue-advanced-cropper` scales the image to *contain* within whatever box it's given by default, so a wide image and a portrait one both letterbox/pillarbox inside the same fixed rectangle instead of resizing the panel — switching between images of different aspect ratios never reflows the layout. `Cropper.vue`'s own `.image-cropper` is `width/height: 100%` (no `max-height` of its own) so that sizing comes purely from this parent box; it also overrides `:deep(.vue-advanced-cropper__background)` to `#ddd` since the library's own letterbox fill is opaque black by default.
- **`src/components/Image/Cropper.vue`** is a thin wrapper around `vue-advanced-cropper`'s `Cropper`, exposing `reset()` and emitting `change` with raw `Coordinates` — it holds no store knowledge.
- **`src/components/Filters/AdjustmentsPanel.vue`** / **`src/components/Filters/FilterPanel.vue`** are controlled, store-agnostic `v-model` components (three sliders; a single-select `v-chip-group` over the filter presets since the store's `filter` slot is single-valued). Each slider in `AdjustmentsPanel.vue` is paired with a narrow `v-text-field` (`type="number"`, two-way synced via `updateFromText`, clamped to `[0, 200]`) for entering an exact value — its native number-spinner is hidden and padding trimmed (see the `:deep(input)` rules) since the default width wasn't enough for 3-digit values without wrapping. Each slider's label is a custom `#label` slot (not the `label` prop) specifically so `@dblclick` can reset that one parameter back to 100 — the `label` prop alone doesn't expose an element to hang the handler off.

## Vuetify configuration (`src/plugins/vuetify.ts`)

- Dark theme is the default — this is a deliberate, domain-informed choice (matches how professional image/print editors default to dark chrome so it doesn't bias brightness/contrast/saturation perception), not a stylistic default. See `NOTES.md` for the full rationale.
- Icons use the **font-based** `vuetify/iconsets/mdi` (+ `@mdi/font`), not `vuetify/iconsets/mdi-svg`. This was a deliberate fix: `mdi-svg` only ships path data for Vuetify's own internal aliases (close, checkbox, etc.), not arbitrary `mdi-*` names like `mdi-crop`/`mdi-pencil`/`mdi-download` used throughout this app — switching iconsets would silently blank out every icon button unless each one is rewired to import `@mdi/js` path constants individually.
- `VSlider`/`VBtn` have global `defaults` configured once here rather than repeating props on every instance.
